## 9. 리팩토링 변경 이력

### 14-1. 추천 로직 1차 리팩토링

- 일자: `2026-04-21`
- 대상: 식단 추천 서비스 전체 흐름

#### 이전 로직

- 추천 과정에서 같은 후보군을 여러 번 다시 조회하고 있었습니다.
- 고칼로리 음식 대체 후보를 찾을 때도 음식별로 대체 후보 조회가 반복되어, 한 번 계산한 결과를 다음 단계에서 재사용하지 못했습니다.
- `highProteinLowFat`, `healthyAlt`, `lowCarb`, `lowFat` 후보군을 여러 번 분산 조회하면서 메모리와 조회 시간이 같이 낭비되는 구조였습니다.

#### 어떻게 바꿨는지

- 추천 흐름에서 한 번 구한 후보군은 같은 요청 안에서 재사용하도록 정리했습니다.
- 고칼로리 음식 대체는 음식마다 개별 후보를 다시 읽지 않고, 대체 후보 풀을 한 번 가져온 뒤 메모리에서 매칭하도록 바꿨습니다.
- `highProteinLowFat`, `healthyAlt`, `lowCarb`, `lowFat` 후보군도 요청당 한 번만 조회하도록 정리했습니다.

#### 측정 결과

- 리팩토링 전
  - Avg: `2523.08 ms`
  - P95: `2575.25 ms`
- 리팩토링 후
  - Avg: `2247.61 ms`
  - P95: `2356.41 ms`
- 개선율
  - Avg `10.91% 감소`
  - P95 `8.50% 감소`

#### 개선 의미

- 추천 전체 경로에서 중복 계산과 중복 조회를 먼저 줄였고, 이후 SQL 병목을 더 줄이기 위한 기반을 만들었습니다.
- 폭이 아주 크지는 않지만, 같은 주제의 다음 최적화와 합쳐졌을 때 누적 성과를 만들기 좋은 1차 정리 단계였습니다.

### 14-2. 추천 음식 SQL 2차 리팩토링

- 일자: `2026-04-21`
- 대상: 추천 음식 랜덤 추출 SQL

#### 이전 로직

- 추천 음식 추출이 `ORDER BY RAND()` 방식으로 되어 있어, 후보 row 수가 커질수록 DB가 전체 대상을 랜덤 정렬한 뒤 일부만 가져오는 구조였습니다.
- `food` 데이터가 `220,004건`인 상태라, 랜덤 추천 한 번을 위해 전체 스캔과 정렬 비용이 크게 발생하고 있었습니다.
- 추천 서비스 1차 리팩토링 이후에도 DB 레벨 병목은 그대로 남아 있었습니다.

#### 어떻게 바꿨는지

- 전체 랜덤 정렬을 없애고, `max food_id`를 먼저 구한 뒤 무작위 pivot id를 기준으로 뒤쪽 후보를 우선 조회하도록 바꿨습니다.
- 뒤쪽 후보만으로 개수가 부족하면 pivot 앞쪽 구간에서 부족한 수만큼 보충해 결과 수를 맞추도록 정리했습니다.
- 즉, 전체 row를 매번 랜덤 정렬하는 대신 PK 범위를 활용해 랜덤 샘플링에 가까운 방식으로 바꿨습니다.

#### 측정 결과

- 리팩토링 전
  - Wall-clock Avg: `639.82 ms`
  - Wall-clock P95: `718.45 ms`
  - Representative actual: `912 ms`
- 리팩토링 후
  - Wall-clock Avg: `296.58 ms`
  - Wall-clock P95: `322.80 ms`
  - Representative actual: `0.115 ms`
- 개선율
  - Wall-clock Avg `53.65% 감소`
  - Wall-clock P95 `55.07% 감소`
  - Representative actual `99.99% 감소`

#### 개선 의미

- 서비스 레벨 중복 제거 다음 단계로, 실제 병목이던 DB 정렬 비용을 직접 줄인 개선입니다.
- 데이터가 커질수록 차이가 더 커질 수 있는 구조라서, 포트폴리오에서 SQL 최적화 성과로 쓰기 좋습니다.

### 14-3. 약 조회 Cache-First 리팩토링

- 일자: `2026-04-21`
- 대상: 약 검색, 약 상세 조회

#### 이전 로직

- 약 검색은 검색어가 이미 로컬 DB에 있어도 먼저 외부 API 경로를 타거나, fallback 과정에서 중복 외부 호출이 발생할 여지가 있었습니다.
- 약 상세는 DB에 주요 상세 텍스트가 저장되어 있어도 다시 외부 API를 호출하는 경로가 남아 있었습니다.
- 검색과 상세 보조 메서드들이 부분적으로 겹치면서, 같은 데이터를 다른 경로로 다시 가져오는 구조가 생겨 있었습니다.

#### 어떻게 바꿨는지

- 검색은 로컬 약 이름 캐시를 먼저 조회하고, 캐시에 결과가 없을 때만 외부 API를 호출하도록 바꿨습니다.
- 상세는 로컬 row에 이름, 효능, 복용법, 주의사항 같은 핵심 설명 텍스트가 이미 있으면 외부 API 호출 없이 즉시 반환하도록 early return을 추가했습니다.
- 보조 상세 조회 흐름을 하나의 공통 경로로 모아, 검색과 상세가 같은 캐시 전략을 쓰도록 정리했습니다.
- 검색 fallback도 불필요한 중복 호출이 생기지 않도록 1회 선택 구조로 단순화했습니다.

#### 측정 결과

### 14-6. 식단 수정 Diff Sync 리팩토링

- 일자: `2026-04-21`
- 대상: 식단 수정 서비스, `PUT /api/diet/{dietId}` with 5 items

#### 이전 로직

- 식단 수정 요청이 들어오면 먼저 상위 식단 row를 수정하고, 기존 `diet_item`을 전부 삭제한 뒤, 요청으로 들어온 item을 다시 전부 insert하는 구조였습니다.
- 실제 변경량이 작아도 매번 전체 삭제 + 전체 재삽입이 발생했습니다.
- 5개 item 수정 시 대표 write 수는 아래와 같았습니다.
  - `diet_log update 1`
  - `diet_item delete all 1`
  - `diet_item insert 5`
  - 총 `7 writes`

#### 어떻게 바꿨는지

- 먼저 기존 item 목록을 읽어온 뒤, 요청 item과 비교해 `유지 / 수정 / 삭제 / 추가`로 분기하는 diff sync 방식으로 바꿨습니다.
- 값이 바뀌지 않은 item은 write를 발생시키지 않고 그대로 유지합니다.
- 값이 바뀐 item만 update하고, 새 item만 insert하며, 요청에서 빠진 기존 item만 delete 하도록 정리했습니다.
- 대표 시나리오는 `3개 unchanged + 1개 update + 1개 delete + 1개 insert` 기준으로 측정했습니다.

#### 측정 결과

- 리팩토링 후 raw times
  - `18.69 ms`
  - `13.57 ms`
  - `21.41 ms`
  - `14.74 ms`
  - `16.23 ms`
- 대표 write 수
  - 리팩토링 전: `7`
  - 리팩토링 후: `4`
- 응답시간
  - Avg: `33.45 ms -> 16.93 ms`
  - P95: `34.76 ms -> 21.41 ms`
- 개선율
  - 대표 write 수 `42.86% 감소`
  - Avg `49.39% 감소`
  - P95 `38.41% 감소`

#### 개선 의미

- 수정 API가 이제 item 전체를 갈아엎는 대신 실제 변경분에 더 가깝게 반응하는 구조가 됐습니다.
- unchanged item이 많을수록 효과가 커지므로, 데이터가 커질수록 체감 차이가 더 커질 가능성이 높습니다.

- 서비스 레이어에서는 세션 의존을 제거하고, 필요한 사용자 식별자를 컨트롤러가 명시적으로 전달하도록 정리했습니다.

- 컨트롤러, 서비스, 스케줄러, DAO에 같은 패턴이 흩어져 있어 주입 방식이 일관되지 않았습니다.

#### 어떻게 바꿨는지

- 필드에 달린 `@Autowired`를 모두 제거하고, 필요한 의존성은 생성자 파라미터로 명시적으로 받도록 전환했습니다.

- `@Qualifier`가 필요한 경우는 생성자 파라미터에 그대로 유지해, 구현체 선택 방식은 보존하면서 주입 방식만 일관되게 바꿨습니다.
- 이미 생성자를 쓰고 있던 클래스는 누락된 의존성만 생성자에 포함시켜 한 가지 주입 방식만 남기도록 정리했습니다.

### 14-11. 복약 로그 중복 저장 방지 리팩토링

- 일자: `2026-04-21`
- 대상: 복약 로그 저장, 복약 순응도 통계와 PDF 리포트의 원천 데이터 정합성

#### 이전 로직

- 복약 로그 저장 시 같은 복약 이벤트를 식별하는 기준이 명확하지 않아, 사용자의 중복 클릭이나 네트워크 재전송이 발생하면 동일한 복약 기록이 여러 번 저장될 수 있는 구조였습니다.
- 복약 로그는 단순 이력 데이터가 아니라 이후 복약 순응도 통계와 PDF 리포트의 원천 데이터이기 때문에, 같은 의미의 이벤트가 중복 저장되면 분석 결과 전체의 신뢰도가 떨어질 수 있었습니다.
- 애플리케이션 레벨 검증만으로 중복을 막는 방식은 동시 요청, 재전송, 예외 상황에서 빠질 수 있어 DB 차원의 마지막 보호막이 필요하다고 판단했습니다.

#### 어떻게 바꿨는지

- 복약 로그 한 건의 유일한 기준을 `약 ID`, `복용 날짜`, `복용 회차` 3개 컬럼 조합으로 정의했습니다.
- 애플리케이션에서는 요청한 사용자가 해당 약의 소유자인지, 복용 회차가 유효 범위인지 검증하도록 정리했습니다.
- DB에는 `(약 ID, 복용 날짜, 복용 회차)` 기준의 복합 유니크 인덱스를 적용해 동일 복약 이벤트가 중복 저장되지 않도록 했습니다.
- 운영 중인 데이터를 고려해 바로 인덱스를 추가하지 않고, `컬럼 추가 -> 기존 데이터 backfill -> NOT NULL 적용 -> 중복 그룹 점검 -> 유니크 인덱스 생성`의 5단계 이행 순서로 정리했습니다.

#### 측정 결과

- 복약 로그 유일성 기준
  - 리팩토링 전: 명확한 단일 기준 없음
  - 리팩토링 후: `약 ID + 복용 날짜 + 복용 회차`
- DB 정합성 보호 장치
  - 리팩토링 전: 애플리케이션 검증 중심
  - 리팩토링 후: 복합 유니크 인덱스 `1개` 추가
- 운영 이행 단계
  - `컬럼 추가`
  - `기존 데이터 backfill`
  - `NOT NULL 적용`
  - `중복 그룹 점검`
  - `유니크 인덱스 생성`

#### 개선 의미

- 복약 로그 한 건의 기준을 3개 컬럼 조합으로 명확히 정의하면서 어떤 데이터가 중복인지 판단할 수 있는 기준이 생겼습니다.
- 중복 방지 책임을 애플리케이션 검증에만 두지 않고 DB 복합 유니크 인덱스까지 확장해, 동일 복약 이벤트가 중복 저장되어 통계나 PDF 리포트가 왜곡될 가능성을 구조적으로 차단했습니다.
- 운영 중 스키마 변경 시 기존 데이터와 충돌하지 않도록 단계적으로 이행해 데이터 정합성과 변경 안정성을 함께 확보했습니다.
